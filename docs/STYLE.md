# 文体スタイルガイド

本ドキュメントの章本文（`docs/*.md` および `docs/appendix-*.md`）を、教科書のように淡々とした文体に揃えるための規則集です。語彙・語尾・構文パターンの3層を扱います。

## このガイドの位置づけ

本ガイドは [WRITING_GUIDE.md](../WRITING_GUIDE.md) の補助です。役割の分担は次のとおりです。

- `WRITING_GUIDE.md`：想定読者像、表記統一、ファイル名規約、Lint運用、章の骨格テンプレートなど、執筆全般の前提
- `docs/STYLE.md`（本書）：文体に紛れ込みがちな「AIっぽさ」のパターンと、その除去方針

`WRITING_GUIDE.md` の「避けたい言い回し」は語彙寄りの代表例集として残し、本書では**用法のアンチパターン**をA〜Fの節構成で扱います。両者で扱いが重なる箇所では、本書を優先します。

## なぜ別建てにするのか

生成AIに下書きを任せた文章には、語彙だけを置換しても残る癖があります。具体的には次の5系統です。

| 系統 | 例 |
| ---- | ---- |
| メタファーの常用 | 「料理番組ふうに」「30年前のメール」「言い切りの強い新人」「机の上の付箋」 |
| 読者への呼びかけ | 「〜と思ってください」「〜と捉えてください」「鋭い読者です」 |
| 口語的なくだけ語 | 「肩慣らし」「ざっくり」「踏む／踏めなければ」「事故りやすい」「うっかり」 |
| マーカー句の常套化 | 「結論を先に一行で言うと」「〜という補助線を引いておくと」「〜という構図になります」 |
| 強調と装飾 | 太字の頻用、「**〜である**」断定、括弧補足の多重化 |

語のブラックリスト方式では、同じ語の機能的な用法と装飾的な用法を分けられません。そこで本書では、語そのものを禁じる代わりに、用法の文脈に踏み込んでアンチパターンを記述します。

## A. 文末・語尾

文体の輪郭は語尾でまず決まります。地の文は事実述語に寄せます。

- 「です・ます」体に統一する
- 命令・依頼系の語尾を原則禁止する。「〜してください」「〜と思ってください」「〜と捉えてください」「〜していただいて構いません」が該当する
- 上記の代わりに事実述語を用いる。「〜である」「〜となる」「〜の場合がある」が該当する
- 「〜にあたります」「〜という形になります」「〜ということになります」を禁止する。`A は B である` で言い切る
- 文末の `？` は本文では使わない。章タイトル・節タイトルには可

例外として、手順セクションでの直接の指示（例：「送信前に内容を確認します」「設定画面で対象を切り替えます」）は宣言形のまま残します。「〜しましょう」「〜してください」を手順節以外で使わない、というのが運用上の境界です。

| Before | After |
| ---- | ---- |
| 〜と思ってください | 〜である |
| 〜と捉えてください | 〜と整理できる |
| 〜にあたります | 〜である |
| 〜という形になります | 〜となる |

## B. 二人称・話者

話者の人格を立てる語と、読者を呼びかけで動かす語を、地の文から落とします。

- 「あなた」「皆さん」「読者」を地の文では使わない。必要なときは「利用者」「担当者」に置き換える
- 「〜していただいた」「〜していただきます」を禁止する
- 一人称（「本ドキュメントは」）は章冒頭の宣言と末尾の参照誘導に限定する

| Before | After |
| ---- | ---- |
| 皆さんが画面で目にしている整った日本語 | 利用者が画面で目にする整った日本語 |
| Geminiに触れていただきました | 1章でGeminiに触れた前提で進めます |
| 鋭い読者です | （削除。本文の側に事実だけ残す） |

## C. 削るべき用法パターン

語のブラックリストではなく、用法のアンチパターンとして扱います。同じ語でも文脈で機能と装飾が分かれます。

### C-1. 平易な動詞の口語的言い換え

| Before | After |
| ---- | ---- |
| URLを踏む | URLを開く |
| APIを叩く | APIを呼び出す |
| プロンプトを投げる | プロンプトを送信する |

「手続きを踏む」のような固定表現は許容します。問題にしたいのは、対象が動詞の本来の意味から離れているにもかかわらず、口語のリズムだけで使う書き方です。

### C-2. 具体名のある事象に当てる包括語

| Before | After |
| ---- | ---- |
| 事故が起きやすい局面 | 情報漏えいが起きやすい局面／誤回答が起きやすい局面 |
| 事故りやすい | 誤操作につながる／権限が想定より広く付与される |

「情報漏えい」「誤操作」「誤回答」「権限過大付与」のように、何の事故かを書き分けます。「事故防止のチェックリスト」のように個別事象が文脈で具体化されている場合は許容します。

### C-3. 読者への語りかけのメタ指示

| Before | After |
| ---- | ---- |
| 〜という補助線を引いておくと、〜が見えてきます | （削除して、補助線の中身を本文の事実として直接書く） |
| 〜という像だけ持っておいてください | 〜である |
| 〜という構図で捉えると | 〜という構造である |
| 〜と思ってください | 〜である |

「XはYを理解するための補助線である」のように、事実述語の中で抽象語を使う形は許容します。読者の理解の進み方を比喩空間で代弁する形を避けます。

### C-4. 装飾としてのカジュアル名詞

| Before | After |
| ---- | ---- |
| 肩慣らしをしていただきました | 1章ではGeminiに触れた |
| ざっくり3ステップ | 大まかに3ステップ |
| 気の利いた返答 | 自然な返答／用件に沿った返答 |
| ぼんやり理解している | 概要は把握している |
| ちらほら出てくる | ところどころ出てくる |
| うっかり共有してしまう | 想定外の相手に共有してしまう |

代替の方向は「大まか」「おおむね」「適切」「的確」のような事務的な語です。

### C-5. 話者キャラ化の語

| Before | After |
| ---- | ---- |
| 鋭い読者です | （削除） |
| うちの会社でも、〜 | （削除して、観察された事実だけを残す） |

事務的な一人称（「本書は〜を扱う」「本ドキュメントでは〜の順で進める」）は可とします。

### C-6. 強意マーカーの常套句

| Before | After |
| ---- | ---- |
| 結論を先に一行で言うと | （削除し、その節の節タイトルを結論にする） |
| 思い切って一文にまとめると | （削除） |
| 〜という構図になります | 〜という構造である |
| 〜にあたります | 〜である |
| 〜という形になります | 〜となる |

文書全体を見出しで構造化しているのに、文中でメタに「いま結論を言います」と宣言する二重化を避けます。

## D. 比喩・たとえ話

比喩そのものを禁じる意図はありません。説明の骨格を担う比喩は採用し、装飾としての比喩は削ります。

### 採用してよい比喩の条件

次の3つを全て満たす比喩は採用します。

1. 目的が明示できる。何を理解させるための比喩か、一文で言える
2. 対応関係が一貫している。A→A'、B→B' を文中で追える
3. 本文の主張に橋を渡している。比喩の含意が次段の説明の補助になる

採用例を示します。

> モデルはAIサービスの頭脳である。頭脳が大量のエネルギーを必要とするように、モデルの稼働にも大量の電力を消費する。

「頭脳」と「エネルギー消費」が対応関係を作り、続く「電力コスト」の説明の補助になっています。

### 禁止する比喩

- 装飾・トーン作りだけの比喩。「料理番組ふうに」「肩慣らし」「土俵に乗せる」「正体をスケッチする」が該当する
- 慣用句化した死んだ比喩で、対応物が文中で示されないもの
- 文化・流行・特定世代に依存する比喩。「30年前のメール」など、目的が明確でも読者層の前提が割れるもの

NG例を示します。

> 料理番組ふうに段階で見ると分かりやすくなります

→ 比喩の対応関係が文中に示されていません。「番組ふう」が指す具体は何か、章を進めるうえで読者に渡せていません。次の段で図解と箇条書きに進めるなら、この一文は削れます。

### 運用ルール

- 比喩を入れる場合は、初出時に対応関係を1文で書く
- 章ごとの個数上限は設けない。基準を満たす限り何度使ってもよい
- 比喩で言い換えただけで前進していない段落（次段の説明と接続しないもの）は削る

## E. 太字の運用

太字は文章での強意で足りない場合に限って使います。語順、文の作り、節タイトルで強調できるなら、まずそちらで処理します。太字は最終手段です。

### 用途は次の2つだけ

1. 定義された用語の初出。章が以後ずっと使う術語を1度だけ太字にする。2度目以降はしない
2. 見出しでは拾えない短い対比語。表や箇条書きの項目名で、対比軸の語そのものを最小限に強調する場合

### 禁止する用法

- 文中の動詞・形容詞・節の太字化（「**こうなります**」「**もっともらしい**」「**毎回もう一度モデルに渡している**」）
- 「ここが重要」を示すための太字。重要なら本文の構造で示す
- 1段落に複数の太字
- 引用ブロック内の太字

### 運用基準

1節につき太字は0〜2箇所が目安です。3箇所を超える場合は、節の組み立てを見直す合図と捉えます。

## F. 段落・節・章のつなぎ

- 節の冒頭で「結論を先に言うと」「一行で言うと」を禁止する。**節タイトル＝結論** にする
- 章末の「まとめ」は箇条書き4点以内、各点は事実1個。「次は〜で扱います」は1文だけ
- 章冒頭の「前章では〜していただきました」式の振り返りは禁止する。代わりに「前提：〜章の内容」を箇条書きで2行に置き換える

章冒頭の例を示します。

| Before | After |
| ---- | ---- |
| 1章で、Geminiと30分ほど肩慣らしをしていただきました | 前提：1章でGeminiのチャット画面に触れている |
| 前章では〜を整理しました。本章では〜を扱います | 前提：前章で〜を整理した／本章では〜を扱う |

## 機械化できる部分（補助）

定型句の一部は禁止語ではなく**要レビュー箇所のヒット**としてgrepで拾えます。対象はC-1の口語動詞、C-3のメタ指示句、C-6の強意マーカーの3群です。下表のパターンで一括検出できます。

### grep パターンの参考表

| パターン | 主に該当する節 | 用途 |
| ---- | ---- | ---- |
| `踏[むみめ]\|叩[くきけ]\|投げ[るたて]` | C-1 | 口語動詞の検出 |
| `事故[るり]\|事故りやすい` | C-2, C-4 | 包括語の検出 |
| `補助線\|像だけ\|構図で捉え\|と思ってください\|と捉えてください` | C-3 | メタ指示の検出 |
| `肩慣らし\|ざっくり\|気の利い\|ぼんやり\|ちらほら\|うっかり` | C-4 | カジュアル名詞の検出 |
| `鋭い読者\|うちの会社` | C-5 | 話者キャラ化の検出 |
| `結論を先に\|一行で言う\|にあたります\|という形になります\|という構図になります` | C-6 | 強意マーカーの検出 |
| `していただ[いきけ]\|あなた\|皆さん` | B | 二人称・呼びかけの検出 |
| `\*\*[^*]+\*\*` | E | 太字の出現箇所の確認（密度チェック用） |

`prh-style.yml` には身体性の比喩・評価ラベル系の検出ルールがすでに整備されています。本書のパターンと合わせ、次の手順で検出できます。

```bash
npm run lint
grep -nE "(踏[むみめ]|叩[くきけ]|投げ[るたて])" docs/*.md
grep -nE "(結論を先に|一行で言う|にあたります|という形になります)" docs/*.md
```

検出はあくまで要レビュー候補の抽出です。最終判断は人の目で行います。

## まとめ

- 文体の癖は語彙だけでなく用法と構文に出るため、A〜Fの3層で扱う
- 比喩と太字は禁じない。採用条件と運用基準を満たす範囲で使う
- 機械的な検出は補助。最終判断は人の目で残す
