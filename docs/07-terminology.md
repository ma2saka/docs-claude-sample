# 7. 用語集: モデル・プロンプト・コンテキストほか

生成AIの記事を読むと、同じ会話を指して「Claudeが答えた」「Anthropicが答えた」「ChatGPTに聞いた」「このモデルが答えた」と、呼び方が頻繁に入れ替わります。これは業界の語彙自体がまだ揺れていることに起因します。本章では、以降の章で前提となる用語を、層の違いがわかる形で整理します。

本章は通読向けにも、必要な項目だけを拾い読みする辞書としても使える構成にしてあります。[1章](01-gemini-in-workspace.md)から[6章](06-hallucination-and-knowledge-literacy.md)まで読み進めた場合は、すでに登場した概念にラベルを貼り直す位置づけになります。

## 対象読者と前提

- 生成AIに関する記事や社内資料を読み、用語の意味を確認したい人
- 他章を読み進める途中で、用語の定義に立ち戻りたくなった人
- [3章](03-multimodal.md)・[4章](04-external-system-integration.md)・[5章](05-misunderstanding-learning.md)・[6章](06-hallucination-and-knowledge-literacy.md)で出てきた「マルチモーダル」「モデル」「コンテキスト」「事前学習」「学習カットオフ」などの語の定義を、同じ場所で揃えて確認したい人

本章で扱う用語は、後続章の節タイトルや表に直接現れます。あらかじめ意味を揃えておくと、各章の議論を同じ座標に置けます。

## モデル・ブランド・プロバイダ・サービス

文章を返している主体を指すとき、層を取り違えると、議論の対象がすれ違います。本ドキュメントでは、以下の4つを区別して使います。

- **モデル** — 文章を実際に生成する本体である。バージョン更新によって、回答の傾向や得意分野まで大きく変わる場合がある
- **モデルのブランド** — 同じ系統のモデル群につけられた呼称である。世代が変わっても据え置かれる場合が多い
- **プロバイダ** — モデルを開発し、APIや自社サービスとして外部に提供している組織である
- **サービス** — 利用者がモデルに依頼を入力するための入口である。チャット画面、デスクトップアプリのほか、内部でAPIを呼ぶ第三者ツールも含まれる

具体例で対応関係を示します。「Claude」はAnthropic社が提供するモデルのブランド名であり、その配下に `Claude Opus`・`Claude Sonnet`・`Claude Haiku` といった具体的なモデルが並びます。利用者が実際に操作するのは、ブラウザで開く `https://claude.ai` やデスクトップアプリといったサービスです。「ChatGPT」もOpenAI社の `GPT` ブランドを内蔵したサービス名であり、モデルやブランドそのものを指す語ではありません。普段「ChatGPTに聞いてみた」と言うとき、実体としてはサービスを指していることになります。

次の表は、4つの層と代表例の対応です（最終確認：2026-04-24）。具体的なモデル名・サービス名はバージョンアップのたびに入れ替わるため、本ドキュメントの記載は執筆時点の例示にあたります。

| 区分 | 例 |
| ---- | ---- |
| プロバイダ | Anthropic、Google、OpenAI |
| モデルのブランド | Claude、Gemini、GPT |
| 具体的なモデル | Claude Opus、Gemini Pro、GPT-5 |
| サービス（プロバイダ自身が運営） | `claude.ai`、Gemini（Google Workspaceのサイドパネルやアプリ）、ChatGPT |
| サービス（第三者が運営） | Cursor、Perplexity、Genspark、OpenRouter |

ブランド名と具体的なモデル名は、ふだんの会話ではブランド名だけで足ります。性能・料金・カットオフ日付など、世代差で結果が変わる話題では、具体的なモデル名まで指定する必要があります。本ドキュメントでも、両者の粒度を意識して使い分けます。

### サービスの2つのかたち

利用者が触れるサービスは、運営主体の違いで2つに分かれます。

- プロバイダ自身が運営するサービス — `claude.ai`、ChatGPT、`gemini.google.com` などである。モデルを開発する会社が直接運営しており、新しいモデルがまず反映される入口にあたる。データの取扱ルールは、各社の利用規約と直結する
- 第三者が運営するサービス — Cursor（コード編集に特化）、Perplexity（検索文脈での回答）、Genspark（探索的な調査）などである。内部でAnthropic・OpenAI・Googleなどのモデルを呼び出し、独自のUIと使い勝手を重ねた構成になっている

第三者サービスの中には、OpenRouterのように、単一のAPIから複数プロバイダのモデルを切り替えて呼び出せる中継サービスもあります。サービス層に位置しつつ、呼び出し先のモデルを利用者が選ぶ設計です。先の表に当てはめると、第三者が運営するサービスのうち、複数プロバイダのモデルをまとめて扱う種類にあたります。

第三者が運営するサービスを業務に持ち込むとき、押さえておきたいのは、サービス側の規約とプロバイダ側の規約が同時に適用される点です。たとえばCursor経由でClaudeを呼ぶ場合、Cursor側のログ保管ルールと、Anthropic側のAPI利用規約の双方がかかります。サービスとモデルの層を分けて捉えておくと、入力データの保管場所や責任の所在を辿る手がかりになります（データ取扱の論点は[9章](09-security-individual.md)で扱います）。

## プロンプト

**プロンプト**は、AIに渡す依頼文を指します。チャット欄に入力する文章そのものが、プロンプトの実体です。

実務上の要点は3つです。

- プロンプトは命令文ではなく依頼文として扱う。前提や条件を補うと、出力の精度が上がりやすい
- 役割・背景・出力形式を分けて書くと、AIが指示を拾いやすくなる
- 長ければよいわけではなく、必要な情報を過不足なく渡すことが結果を左右する

具体的な書き方は、[8章](08-common-capabilities.md)以降の各章で能力や場面とあわせて取り上げます。業務で使えるテンプレートと組み立てパターンは[Appendix: プロンプトの組み立て方](appendix-prompting.md)にまとめてあります。本章ではプロンプト＝AIへの依頼文という定義のみ押さえれば、以降の章を読み進められます。

## システムプロンプト

**システムプロンプト**は、サービス側が会話の前段にあらかじめ差し込んでいる指示文を指します。役割（応答するときに引き受ける立場）、語調、避けるべき応答、参照すべき内部情報といった内容を含み、利用者には表示されないままモデルに渡されています。

本章の「コンテキスト」節で挙げる4つの構成要素のうちの1つにあたります。利用者がそのつど入力する依頼文と並んで毎回コンテキストへ合流するため、同じモデルを呼び出していても、サービスごとに応答の傾向は変わります。その差の多くは、各サービスのシステムプロンプトの違いに由来します。

「学習」と呼ばれがちな経路の整理の中での位置づけは[5章](05-misunderstanding-learning.md)で扱います。サービス間の応答傾向の差を踏まえた使い分けは[8章](08-common-capabilities.md)で扱います。本章では、システムプロンプト＝サービス側が会話の前段に差し込む指示文、という定義のみ示します。

## コンテキスト

**コンテキスト**は、AIがいまの回答を生成するために参照している情報の総体を指します。具体的には、次の4つがコンテキストの構成要素です。

- 今回入力したプロンプト
- その会話のこれまでのやり取り（直前のQ&Aを含む）
- システムプロンプトと呼ばれる、サービス側があらかじめ仕込んだ指示
- アップロードしたファイルや、コネクタ経由で差し込まれた資料

コンテキストには上限があります。この上限を**コンテキストウィンドウ**と呼び、モデルごとに値が決まっています。コンテキストウィンドウを超える分量を入力した場合、超過した部分は参照対象から外れます。送ったはずの末尾が回答に反映されない、といった現象は、ほとんどがこの超過に起因します。

机の上の資料に置き換えると、対応関係は次のように整理できます。コンテキストが机の上の資料に相当し、机の広さがコンテキストウィンドウに相当します。机に乗り切らない分量は読み込みようがない、という比喩がそのまま、コンテキストウィンドウを超えた情報は参照されない、という性質に対応します。コンテキストウィンドウの実数は、本ドキュメントでは[11章](11-gemini-advanced.md)・[13章](13-claude.md)で具体的なモデルとともに扱います。

## アーティファクト

**アーティファクト**は、チャットの会話とは別の領域に成果物を切り出して編集できる仕組みを指します。議事録の下書き、コードの断片、プレビューできるHTMLなどが典型的な対象です。

ClaudeではArtifacts、GeminiではCanvasという名称で提供されていますが、基本の考え方は共通です。会話の流れとは独立した作業領域に成果物を置き、「ここを直して」と指し示しながら推敲を重ねられます。

業務での使いどころと注意点は[8章](08-common-capabilities.md)で扱います。本章では、アーティファクト＝会話から独立した成果物の編集領域、という定義のみ示します。

## トークン

**トークン**は、AIが文章を扱うときの最小単位です。粒度の目安としては、英単語ひとつがおおむね1トークン、日本語は文字種によって2〜3文字で1トークン程度です。普段の利用では、この粒度がトークン数を概算するときの目安になります。

トークンという語が頻繁に出てくる理由は、主に2つあります。

- 課金の単位 — APIから利用する場合、入力トークン数と出力トークン数を基準に料金が決まる
- 上限の単位 — コンテキストウィンドウも「何トークンまで入るか」で表される

業務利用では、トークンの実数を意識する場面は多くありません。ただし「資料を全文投げたら途中で切れた」「APIの請求が想定より高い」といった事態に当たったとき、原因の切り分けはトークンという単位に立ち戻ると進みます。

## Transformerとアテンション

**Transformer**は、現行のLLMの土台となるニューラルネットワークの構造です。2017年に提案され、それまで言語処理の主流であったRNN・LSTMに代わって、その後のモデル設計の中心になりました。

その中核に置かれているのが**アテンション**（self-attention、自己注意）と呼ばれる仕組みです。文章を処理するとき、各トークンが文脈中の他のどの位置をどれくらい参照するかを動的に重みづけし、次のトークンの確率分布の計算につなげます。これによって、長い文脈の中で離れた位置にある語どうしの関係も計算に反映されます。

業務利用の範囲では、内部の数式や層構成まで追う必要はありません。記事や社内資料で出てくる「Transformerベース」「アテンション機構」という表現は、[2章](02-what-is-generative-ai.md)で整理した次トークン予測の計算を支える土台を指しています。Transformer以外の構造も研究されていますが、現行の主要なLLMはおおむねTransformerを基盤にしています。

## 推論モデルと思考モード

**推論モデル**（reasoning model）は、応答を返す前に内部で長めに考えてから回答を組み立てる動作モードを指します。同じ機能は**思考モード**とも呼ばれます。同じLLMの上で、内部の思考にあたるトークン列を多めに生成してから利用者向けの回答を出力する仕組みです。回答までの時間と費用は増える代わりに、多段の数学・コード生成・厳密な手順の組み立てといった題材で精度が上がりやすい、という観察が共有されています。

呼び名は各社で異なります。

| プロバイダ／モデル | 呼び方の例 |
| ---- | ---- |
| Anthropic（Claude） | extended thinking／adaptive thinking（Opus 4.8） |
| Google（Gemini） | Deep Think（Gemini 3.1） |
| OpenAI | `o`系列などの推論モデル |

業界全体としては、推論モデル／思考モードという呼び方が定着しつつあります。記事や社内資料に並ぶ各社の呼称は、いずれも同じ系統の機能を指しており、各社の比較記事もこの対応関係に立って書かれています。Gemini側の世代名については、Gemini 3.1 Deep Thinkが現時点の具体例にあたります（最終確認：2026-05-16）。Claude側では、2026年5月にGAされたClaude Opus 4.8が現時点の具体例にあたり、`adaptive thinking` の方式が採用されています（最終確認：2026-06-11）。具体的なモデル並びでの位置づけは[11章](11-gemini-advanced.md)・[13章](13-claude.md)で扱います。

## 事前学習

**事前学習**（pretraining）は、大量の文章を読み込ませてモデル本体を一から作り上げる学習段階を指します。インターネット上の公開テキストや書籍、コードなどを学習データとし、文章の統計的なパターンをモデルの重みに刻み込みます。

実行するのはモデルを開発するプロバイダ（Anthropic、Google、OpenAIなど）に限られます。期間は数か月から年単位にわたり、大量の計算資源を投じて行われます。業務利用の通常フローで、利用者がチャットに打ち込んだ発話が事前学習の対象になることはありません。

「学習」と呼ばれがちな3区分（事前学習・ファインチューニング・コンテキスト／メモリ）のうち、事前学習はモデル本体の重みを一から作る段階に位置します。学習カットオフは事前学習に使われたデータの終端日を指す語であり、ファインチューニングは事前学習を済ませた既存のモデルに追加データを重ねる調整にあたります。コンテキスト・メモリ・RAGの系統は、モデル本体の重みには触れず、外側から情報を差し込む経路で、別系統に位置します。3区分の整理と業務での扱い方は[5章](05-misunderstanding-learning.md)で扱います。本章では、事前学習＝モデル本体を一から作り上げる学習段階、という定義のみ示します。

## 学習カットオフ

**学習カットオフ**（knowledge cutoff）は、モデルの事前学習に使われたデータが集められた期間の終端日を指します。カットオフ以後に起きた出来事や、その日以降に公開された資料は、モデルそのものの知識には含まれていません。

カットオフはモデルごとに値が異なり、世代の更新で動きます。新しい世代に切り替わったからといって、直近の出来事まで自動的に含まれるとは限りません。実数は各社の公式ドキュメントを一次ソースとして確認します。主要モデルの参照先は[6章](06-hallucination-and-knowledge-literacy.md)に表でまとめてあります。

カットオフ以後の情報を扱うときは、会話のコンテキストに後から差し込む経路を取ります。添付資料、Webコネクタの取得結果、社内文書を引くRAGなどがこれにあたります。位置づけと典型的な使い分けは[6章](06-hallucination-and-knowledge-literacy.md)、社内文書の参照経路は[Appendix: 拡張検索と埋め込み検索](appendix-retrieval-and-embeddings.md)で扱います。

## ファインチューニング

**ファインチューニング**（fine-tuning）は、事前学習を済ませた既存のモデルに追加の教材を読ませて派生版を作る手続きを指します。法務文書や医療文書のように、特定のドメインに合わせた応答が必要な場面で選ばれます。

学習カットオフがモデルの事前学習に紐づく終端日を指すのに対し、ファインチューニングは事前学習の上に追加の調整を重ねる動作にあたります。モデル本体の重みを更新する点で事前学習と同じ系統に位置し、コンテキストやRAGのように重みは変えずに情報を外側から差し込む経路とは別の系統です。プロバイダ側で行う場合と、利用者が自社データを使ってAPI経由で行う場合があり、業務利用の通常フローでチャットの内容が自動的に派生版へ反映されることはありません。

「学習」と呼ばれがちな3区分（事前学習・ファインチューニング・コンテキスト／メモリ）の中での位置づけと、業務での扱い方は[5章](05-misunderstanding-learning.md)で扱います。提供事業者のクラウドではなく自社内や手元のPCでモデルを動かす場合のファインチューニングは、[Appendix: ローカルLLM](appendix-local-llm.md)で扱います。本章では、ファインチューニング＝既存モデルに追加データを読ませて派生版を作る手続き、という定義のみ示します。

## RAG

**RAG**（Retrieval-Augmented Generation、検索拡張生成）は、質問に関連する文書を検索で取り出し、その内容をプロンプトに合流させてから回答を組み立てる段取りの呼び名です。「質問→検索→関連文書をプロンプトに添える→回答」という3段で骨格が成り立っており、特定の製品や個別の実装手法を指す語ではありません。

学習カットオフ以後の情報や、そもそも学習データに含まれていない社内文書を扱う場面で選ばれます。モデル本体には手を加えず、コンテキストに毎回必要な分だけを差し込む経路にあたり、本章の「コンテキスト」節で整理した4つの構成要素のうち「アップロードしたファイルや、コネクタ経由で差し込まれた資料」の系統を拡張したものに位置づけられます。社内ナレッジ検索のチャットUIや「ファイルをアップロードして質問する」機能の内側でも、同じ段取りが動いている場合がほとんどです。

なお、Geminiの文脈では同じ骨格をグラウンディング（Grounding）という呼称で扱う場面が多くあります。検索方式（キーワード検索／埋め込み検索／ハイブリッド検索）の選び方や、ベンダー提案を読み解くための整理は[Appendix: 拡張検索と埋め込み検索](appendix-retrieval-and-embeddings.md)で扱います。本章では、RAG＝検索結果をプロンプトに合流させる段取りの呼び名、という定義のみ示します。

## グラウンディング

**グラウンディング**（Grounding）は、GoogleがGeminiの文脈で、外部情報を参照しながら回答する機能の総称として用いている呼称です。代表例が「Grounding with Google Search」で、Gemini APIから有効化すると、モデルが検索結果を踏まえて回答を返し、引用URLや検索クエリも合わせて返します。

直前のRAG節で扱った段取りと、仕組みのレベルではほぼ重なります。外部から取り出した情報を生成のプロンプトに合流させて回答を組み立てる、という骨格は同じです。違いは強調点と想定する情報源にあります。RAGは学術用語として「検索した文書を生成の素材として使う」段取りの呼び名で、対象とする情報源は社内文書・Web・データベースなど任意の外部記憶に及びます。グラウンディングはGoogleが製品文脈で採用した呼称にあたり、「回答を事実に接地（ground）させる」側面が前面に出ます。想定する情報源としてGoogle検索やGoogle Mapsが具体例として並びます。

RAG・Embedding・ベクトル検索との対応関係や、検索方式の整理は[Appendix: 拡張検索と埋め込み検索](appendix-retrieval-and-embeddings.md)で扱います。本章では、グラウンディング＝Geminiの文脈でRAGの骨格を呼び直した呼称、という定義のみ示します。

## Embedding

**Embedding**（埋め込み）は、文章を変換した結果の高次元ベクトル（数百〜数千個の数字の並び）を指す呼び名です。変換を担うモデルは「埋め込みモデル」（Embedding model）と呼びます。ベクトル同士の距離が近いほど意味が近いとみなす検索方式の素地として用いられます。

直前のRAG節・グラウンディング節で扱った段取りは、外部から取り出した情報を生成のプロンプトに合流させる枠組みでした。その内側で「関連する文書を取り出す」検索の部分を担う代表的な実装手段が、Embeddingを使った検索（埋め込み検索）です。意味の近い文章はベクトル上でも近い位置に配置されるため、語彙の違う言い換えや同義語にも反応する検索が成り立ちます。一方で、固有名詞や型番のピンポイント検索は苦手なため、業務利用では従来のキーワード検索と組み合わせるハイブリッド構成を選ぶ場面が多くなります。

埋め込みモデルの代表例、ベクトル検索とハイブリッド検索の組み立て、ベンダー提案を読み解く際の整理軸は[Appendix: 拡張検索と埋め込み検索](appendix-retrieval-and-embeddings.md)で扱います。本章では、Embedding＝文章を変換した結果の高次元ベクトル（変換を担うモデルは埋め込みモデル）、という定義のみ示します。

## メモリ

**メモリ**は、利用者の好みや事実情報を会話をまたいで長期間保持し、新しい会話を始めるたびにコンテキストへ自動的に差し込む仕組みを指します。Claudeでは「メモリ」、Geminiでは「保存済み情報」という呼称で提供されています。

[5章](05-misunderstanding-learning.md)で整理した「学習」の3区分のうち、事前学習・ファインチューニングはモデル本体の重みに残る系統です。これに対し、コンテキストとメモリは重みには残らず、モデルの外側からその都度差し込まれる系統に位置します。直前の節で扱ったRAGも同じ「外側から差し込む」系統に含まれます。RAGが文書集合から関連分を検索で取り出して差し込む経路であるのに対し、メモリは利用者個人に紐づく好みや事実情報を保存しておき、新しい会話を開くたびにコンテキストへ自動的に差し込まれる経路です。

具体的なサービスごとの使い方は、Claude側を[13章](13-claude.md)、Gemini側を[11章](11-gemini-advanced.md)で扱います。保存内容の棚卸しや削除の手順は、[9章](09-security-individual.md)の「履歴とメモリ」節にまとめてあります。本章では、メモリ＝利用者の文脈を会話をまたいで保持し、新しい会話で自動的に差し込まれる仕組み、という定義のみ示します。

## マルチモーダル

**マルチモーダル**は、テキスト・画像・音声・動画など、複数の様式（モダリティ）の情報を1つのモデルで一緒に扱える性質を指します。「マルチ」（複数の）と「モーダル」（様式）を組み合わせた語です。

2026年時点では、Claude・Gemini・ChatGPTなど主要サービスの標準のチャット画面で、文章のほかに画像やPDFを入力として受け取り、回答に組み込む使い方が実用域に入っています。画像ならピクセルの並び、音声なら音の波形を、モデルが扱える共通の表現に変換する前段が内部に置かれており、その先の生成処理は文章のときと近い形で動きます。

入力で扱える素材は画像・PDF・音声・動画の4系統、出力は画像生成・動画生成・音声合成・音声対話の4系統に整理できます。業務での具体的な使いどころと、各社の対応プラン・権利・表示の取り扱いは[3章](03-multimodal.md)で扱います。本章では、マルチモーダル＝複数の様式を共通の表現に揃えて扱える性質、という定義のみ示します。

## ハルシネーション

**ハルシネーション**は、生成AIがもっともらしい体裁のまま事実と異なる内容を出力する現象を指します。存在しない書籍の引用、実在の人物への架空の経歴の付与、根拠のない数字の提示など、文体は自然なまま中身が誤っている、という点に特徴があります。

[2章](02-what-is-generative-ai.md)で扱った、次のトークンを確率で選ぶ計算から直接導かれる性質です。モデルが最適化されているのは、自然な文章を組み立てる方向です。出力内容の真偽は、それとは別の評価軸にあたります。自然な体裁のまま、事実関係に誤りのある応答が返る場面も残ります。

業務利用での扱い方と、出力の裏取りの段取りは[6章](06-hallucination-and-knowledge-literacy.md)で独立した章として整理しています。本章では、ハルシネーション＝もっともらしい体裁のまま事実誤認が混ざる現象、という定義のみ示します。

## Web検索

**Web検索**は、サービス内蔵のチャットUIから公開Webページを検索し、検索結果を踏まえて回答を組み立てる仕組みの共通呼称です。利用者が依頼に「最新の〜を調べて」のような時事性の語を含めると、サービス側で検索クエリを組み立て、いくつかのページを参照したうえで回答を返します。回答にソースURLが添えられる場合も多く、参照されたページを利用者の側で確かめられます。

仕組みのレベルでは、本章の「グラウンディング」節で扱った機能と大きく重なります。同節で代表例として示した「Grounding with Google Search」は、利用者から見るとWeb検索モードにあたります。Claude・ChatGPTも同じ系統の機能を「Web検索」の呼び名で提供しています。続く「コネクタ」節で扱う仕組みは特定のSaaS（Slack・Google Workspaceなど）との接続経路を指すのに対し、Web検索は公開Webページ全般を情報源とします。

裏取りの出発点としての位置づけは[6章](06-hallucination-and-knowledge-literacy.md)、Claudeでの動き方とResearch・Deep Researchとの使い分けは[13章](13-claude.md)で扱います。本章では、Web検索＝サービス内蔵で公開Webを情報源にする回答機能の共通呼称、という定義のみ示します。

## Research・Deep Research

**Research・Deep Research**は、調査テーマからまとまったレポートを得る機能の共通呼称です。サービス側が複数のWebソースを自律的に探索し、引用元つきの構造化されたレポートにまとめて返します。応答までには数分から数十分を要します。Claudeでは「Research」、Geminiでは「Deep Research」と呼ばれます。本ドキュメントでは両者を同じ系統の機能として扱います。

直前の「Web検索」節が扱う機能は、会話の流れの中で都度情報を補う軽量な仕組みでした。Research・Deep Researchは、テーマから調査レポートが返るまでを1単位として扱う派生にあたります。応答までの時間は伸びますが、複数ソースの横断的な整理と引用元の明示が標準で含まれます。内側ではWeb検索を含むツール呼び出しが連鎖して進みます。本章で続く「エージェント」節で扱う、AIが連続して外部操作を進めるモードと同じ系統に位置します。

出力されるレポートは下書きの扱いです。引用元のリンクが添えられていても、リンクが付いていることと内容の正確性は別の評価軸にあたります。業務利用では、[6章](06-hallucination-and-knowledge-literacy.md)で整理した一次ソースでの裏取り手順を経由してから、社内・社外の文書として整えます。

Claudeでの利用者から見える流れ・Google Workspace接続時の挙動・Web検索との対比は[13章](13-claude.md)、Geminiでの調査計画の提示と承認の手順・利用条件は[11章](11-gemini-advanced.md)で扱います。本章では、Research・Deep Research＝複数のWebソースを自律的に探索して引用元つきレポートにまとめる機能の共通呼称、という定義のみ示します。

## ツール呼び出し

**ツール呼び出し**（tool use、function calling）は、AIが外部の機能を使うための標準的な約束ごとを指します。AI本体はネットワークに出ず、外側のプログラムを介して外部と接続します。

使える道具の一覧を前もって渡しておくと、AIは依頼内容に応じて「この道具を、この引数で呼び出す」という構造化された依頼文を出力します。外側のプログラム（クライアント）がその依頼文を読み取り、実際のAPIやコマンドを実行して結果をAIに戻します。AIはその結果を踏まえて、人間向けの応答にまとめ直します。

本章で続く節のうち、コネクタ・MCP・エージェント・プロンプトインジェクションは、いずれもツール呼び出しを基盤とする応用にあたります。コネクタはサービス提供者があらかじめ用意したツール呼び出しの経路です。MCPはツール呼び出しを共通プロトコルとして標準化した規格です。エージェントは、ツール呼び出しを連続して進めて目的の成果物までたどり着くモードを指します。プロンプトインジェクションは、外部から取り込んだ文章に紛れた指示が、依頼の延長としてツール呼び出しを誘導する現象です。

仕組みの4ステップと、コネクタ・MCP・自前ツールといった経路ごとの違いは[4章](04-external-system-integration.md)で扱います。ツール呼び出しが外部システムを能動的に操作する場面で生じるガバナンス上の論点は[10章](10-security-agent-era.md)で扱います。本章では、ツール呼び出し＝AIが構造化された依頼文を出力し、外側のプログラムが実行して結果をAIに戻す段取り、という定義のみ示します。

## コネクタ

**コネクタ**は、サービス提供者があらかじめ用意した、外部サービスとの接続経路を指します。利用者は設定画面でスイッチを有効にするだけで、AIがメール・カレンダー・ドキュメントなどの情報を参照できるようになります。

内部の仕組みは[4章](04-external-system-integration.md)で扱ったツール呼び出しそのものです。道具の一覧と認証をサービス提供者側が用意している点が、自前のAPI連携やMCPとの違いにあたります。

コネクタの有効化は、その範囲の情報がAIのコンテキストに入りうる経路を1つ増やすことを意味します。セキュリティ面の判断は[9章](09-security-individual.md)で扱います。

## MCP（Model Context Protocol）

**MCP**は、AIからのツール呼び出しを共通プロトコルとして標準化した規格です。Anthropicが2024年に公開し、現在は他社のクライアントでも対応が進んでいます。

コネクタがサービス提供者の用意した既製の経路であるのに対し、MCPは規格に沿ってツールを自作・追加できる点が異なります。1つのMCPサーバを立てれば、対応する複数のクライアントから同じ作法で呼び出せるため、ツールの再利用がしやすい構造です。

仕組みの詳細は[4章](04-external-system-integration.md)、実際の使いどころは[Appendix: Claude Code](appendix-claude-code.md)で扱います。

## エージェント

**エージェント**は、AIがチャットで答えるだけにとどまらず、目的を達成するためにツール呼び出しや外部操作まで自分で進めるモードを指します。具体的な動作には、外部APIの呼び出しやファイルの読み書きが含まれ、場合によってはブラウザの操作にまで及びます。

従来のチャットとの違いを整理すると、以下のようになります。

| 観点 | 従来のチャット | エージェント |
| ---- | ---- | ---- |
| ユーザーの関与 | 一問一答のたびに人が判断する | 途中の判断はAIが担う |
| 外部操作 | 原則なし（または確認つき） | AIが能動的にツールを呼ぶ |
| 成果物 | 会話の返答 | タスクの完遂（ファイル、投稿、予約など） |

エージェントを使うと、AIが利用者の確認を都度経ずに外部のシステムを操作する場面が生まれます。この点に伴うセキュリティ上の論点は[10章](10-security-agent-era.md)で扱います。

## プロンプトインジェクション

**プロンプトインジェクション**は、外部から取り込んだ文章に紛れた指示が、利用者の依頼より優先される形でエージェントに実行される現象を指します。Webページの本文や社内文書、画像内の文字情報まで、コンテキストへ合流し得る素材であれば経路になります。

[2章](02-what-is-generative-ai.md)で扱った次トークン予測の計算は、コンテキストに載った文字列を手がかりに次の一手を決めます。モデル側では、ユーザーから受け取った依頼と、外部入力に紛れた指示を内部で明確に区別する仕組みは持ちません。公開ページや添付資料に仕込まれた指示文がエージェントの作業中にコンテキストへ合流すると、依頼の延長として処理される瞬間があります。

具体的な経路、利用者の側で取れる対策、組織側の権限分離の考え方は[10章](10-security-agent-era.md)で独立した節として扱います。本章では、プロンプトインジェクション＝外部入力に紛れた指示が依頼より優先されて作用する現象、という定義のみ示します。

## まとめ

- モデル・ブランド・プロバイダ・サービスは層が異なる。ChatGPTや `claude.ai` はサービスの名称であり、モデルやブランドの名称ではない
- 第三者が運営するサービスを業務に持ち込む場合、サービス側の規約とプロバイダ側の規約は同時に適用される
- プロンプトとコンテキストはセットで定義が決まるため、ペアで参照すると意味の輪郭が安定する
- トークンは課金・上限のいずれにも基準として使われ、料金と入力サイズの話題で参照される

次は[8章（生成AIでできること 共通編）](08-common-capabilities.md)で、ここまでの用語を前提に、ClaudeとGeminiが共通して提供する能力をチャット・アーティファクト・コネクタの3本柱として整理します。
