# 7. 用語集: モデル・プロンプト・コンテキストほか

生成AIの記事を読むと、同じ会話を指して「Claudeが答えた」「Anthropicが答えた」「ChatGPTに聞いた」「このモデルが答えた」と、呼び方が頻繁に入れ替わります。これは業界の語彙自体がまだ揺れていることに起因します。本章では、以降の章で前提となる用語を、層の違いがわかる形で整理します。

本章は通読向けにも、必要な項目だけを拾い読みする辞書としても使える構成にしてあります。[1章](01-gemini-in-workspace.md)から[6章](06-hallucination-and-knowledge-literacy.md)まで読み進めた場合は、すでに登場した概念にラベルを貼り直す位置づけになります。

## 対象読者と前提

- 生成AIに関する記事や社内資料を読み、用語の意味を確認したい人
- 他章を読み進める途中で、用語の定義に立ち戻りたくなった人
- [4章](04-external-system-integration.md)・[5章](05-misunderstanding-learning.md)・[6章](06-hallucination-and-knowledge-literacy.md)で出てきた「モデル」「コンテキスト」「学習カットオフ」などの語の定義を、同じ場所で揃えて確認したい人

本章で扱う用語は、後続章の節タイトルや表に直接現れます。あらかじめ意味を揃えておくと、各章の議論を同じ座標に置けます。

## モデル・ブランド・プロバイダ・サービス

文章を返している主体を指すとき、層を取り違えると、議論の対象がすれ違います。本ドキュメントでは、以下の4つを区別して使います。

- **モデル** — 文章を実際に生成する本体である。バージョン更新によって、回答の傾向や得意分野まで大きく変わる場合がある
- **モデルのブランド** — 同じ系統のモデル群につけられた呼称である。世代が変わっても据え置かれる場合が多い
- **プロバイダ** — モデルを開発し、APIや自社サービスとして外部に提供している組織である
- **サービス** — 利用者がモデルに依頼を入力するための入口である。チャット画面、デスクトップアプリのほか、内部でAPIを呼ぶ第三者ツールも含まれる

具体例で対応関係を示します。「Claude」はAnthropic社が提供するモデルのブランド名であり、その配下に `Claude Opus`・`Claude Sonnet`・`Claude Haiku` といった具体的なモデルが並びます。利用者が実際に操作するのは、ブラウザで開く `https://claude.ai` やデスクトップアプリといったサービスです。「ChatGPT」もOpenAI社の `GPT` ブランドを内蔵したサービス名であり、モデルやブランドそのものを指す語ではありません。普段「ChatGPTに聞いてみた」と言うとき、実体としてはサービスを指していることになります。

次の表は、4つの層と代表例の対応です（最終確認：2026-04-24）。具体的なモデル名・サービス名はバージョンアップのたびに入れ替わるため、本ドキュメントの記載は執筆時点の例示にあたります。

| 区分 | 例 |
| ---- | ---- |
| プロバイダ | Anthropic、Google、OpenAI |
| モデルのブランド | Claude、Gemini、GPT |
| 具体的なモデル | Claude Opus、Gemini Pro、GPT-5 |
| サービス（プロバイダ自身が運営） | `claude.ai`、Gemini（Google Workspaceのサイドパネルやアプリ）、ChatGPT |
| サービス（第三者が運営） | Cursor、Perplexity、Genspark、OpenRouter |

ブランド名と具体的なモデル名は、ふだんの会話ではブランド名だけで足ります。性能・料金・カットオフ日付など、世代差で結果が変わる話題では、具体的なモデル名まで指定する必要があります。本ドキュメントでも、両者の粒度を意識して使い分けます。

### サービスの2つのかたち

利用者が触れるサービスは、運営主体の違いで2つに分かれます。

- プロバイダ自身が運営するサービス — `claude.ai`、ChatGPT、`gemini.google.com` などである。モデルを開発する会社が直接運営しており、新しいモデルがまず反映される入口にあたる。データの取扱ルールは、各社の利用規約と直結する
- 第三者が運営するサービス — Cursor（コード編集に特化）、Perplexity（検索文脈での回答）、Genspark（探索的な調査）などである。内部でAnthropic・OpenAI・Googleなどのモデルを呼び出し、独自のUIと使い勝手を重ねた構成になっている

第三者サービスの中には、OpenRouterのように、単一のAPIから複数プロバイダのモデルを切り替えて呼び出せる中継サービスもあります。サービス層に位置しつつ、呼び出し先のモデルを利用者が選ぶ設計です。先の表に当てはめると、第三者が運営するサービスのうち、複数プロバイダのモデルをまとめて扱う種類にあたります。

第三者が運営するサービスを業務に持ち込むとき、押さえておきたいのは、サービス側の規約とプロバイダ側の規約が同時に適用される点です。たとえばCursor経由でClaudeを呼ぶ場合、Cursor側のログ保管ルールと、Anthropic側のAPI利用規約の双方がかかります。サービスとモデルの層を分けて捉えておくと、入力データの保管場所や責任の所在を辿る手がかりになります（データ取扱の論点は[9章](09-security-individual.md)で扱います）。

## プロンプト

**プロンプト**は、AIに渡す指示文を指します。チャット欄に入力する文章そのものが、プロンプトの実体です。

実務上の要点は3つです。

- プロンプトは命令文ではなく依頼文として扱う。前提や条件を補うと、出力の精度が上がりやすい
- 役割・背景・出力形式を分けて書くと、AIが指示を拾いやすくなる
- 長ければよいわけではなく、必要な情報を過不足なく渡すことが結果を左右する

よいプロンプトの書き方は[8章](08-common-capabilities.md)で扱います。本章ではプロンプト＝AIへの依頼文という定義のみ押さえれば、以降の章を読み進められます。

## コンテキスト

**コンテキスト**は、AIがいまの回答を生成するために参照している情報の総体を指します。具体的には、次の4つがコンテキストの構成要素です。

- 今回入力したプロンプト
- その会話のこれまでのやり取り（直前のQ&Aを含む）
- システムプロンプトと呼ばれる、サービス側があらかじめ仕込んだ指示
- アップロードしたファイルや、コネクタ経由で差し込まれた資料

コンテキストには上限があります。この上限を**コンテキストウィンドウ**と呼び、モデルごとに値が決まっています。コンテキストウィンドウを超える分量を入力した場合、超過した部分は参照対象から外れます。送ったはずの末尾が回答に反映されない、といった現象は、ほとんどがこの超過に起因します。

机の上の資料に置き換えると、対応関係は次のように整理できます。コンテキストが机の上の資料に相当し、机の広さがコンテキストウィンドウに相当します。机に乗り切らない分量は読み込みようがない、という比喩がそのまま、コンテキストウィンドウを超えた情報は参照されない、という性質に対応します。コンテキストウィンドウの実数は、本ドキュメントでは[11章](11-gemini-advanced.md)・[13章](13-claude.md)で具体的なモデルとともに扱います。

## トークン

**トークン**は、AIが文章を扱うときの最小単位です。粒度の目安としては、英単語ひとつがおおむね1トークン、日本語は1文字あたり1〜2トークン程度に変換されることが多く、モデルやトークナイザによって増減します。普段の利用ではこの程度の感覚で差し支えありません。

トークンという語が頻繁に出てくる理由は、主に2つあります。

- 課金の単位 — APIから利用する場合、入力トークン数と出力トークン数を基準に料金が決まる
- 上限の単位 — コンテキストウィンドウも「何トークンまで入るか」で表される

業務利用では、トークンの実数を意識する場面は多くありません。ただし「資料を全文投げたら途中で切れた」「APIの請求が想定より高い」といった事態に当たったとき、トークンという単位を知っておくと、原因の切り分けが進めやすくなります。

## 基盤モデルとTransformer

**基盤モデル**（foundation model）は、大量かつ多様なデータで一括して訓練し、要約・翻訳・コード生成・画像理解などの幅広い下流タスクへ転用できる大規模AIモデルの総称です。LLMは、その中で言語処理を主に担う系統にあたります。画像生成や音声認識のモデルも、基盤モデルとして括られる場合があります。

**Transformer**は、現代のLLMが内部で採用しているニューラルネットワーク構造の名称です。入力された各トークンの間の関連性を**アテンション**（attention）と呼ばれる仕組みで重み付けし、文脈に応じた出力を組み立てます。本ドキュメントの本文ではここまで踏み込みませんが、解説記事や論文を読むときに頻繁に登場するため、語として押さえておきます。

## 推論モード（Extended Thinking／長考モード）

**推論モード**は、応答を返す前に、モデル内部で推論ステップを重ねるよう調整された動作モードです。Claudeでは「Extended Thinking」、Geminiでは長考モードや`thinking`系の名称、ChatGPTでは「reasoning」系のモデルで提供されます。

多段の論理を要する依頼や、コードや数式の組み立てを伴う依頼で、このモードを有効にしておくと、応答の精度を安定させやすくなります。応答までの時間とトークン消費は、推論モードを使わない場合より増えます。本ドキュメントでは、利用場面の判断に直結する箇所（[13章](13-claude.md)など）で再登場します。

## エージェント

**エージェント**は、AIがチャットで答えるだけにとどまらず、目的を達成するためにツール呼び出しや外部操作まで自分で進めるモードを指します。具体的な動作には、外部APIの呼び出しやファイルの読み書きが含まれ、場合によってはブラウザの操作にまで及びます。

従来のチャットとの違いを整理すると、以下のようになります。

| 観点 | 従来のチャット | エージェント |
| ---- | ---- | ---- |
| ユーザーの関与 | 一問一答のたびに人が判断する | 途中の判断はAIが担う |
| 外部操作 | 原則なし（または確認つき） | AIが能動的にツールを呼ぶ |
| 成果物 | 会話の返答 | タスクの完遂（ファイル、投稿、予約など） |

エージェントを使うと、AIが利用者の確認を都度経ずに外部のシステムを操作する場面が生まれます。この点に伴うセキュリティ上の論点は[10章](10-security-agent-era.md)で扱います。

## まとめ

- モデル・ブランド・プロバイダ・サービスは層が異なる。ChatGPTや `claude.ai` はサービスの名称であり、モデルやブランドの名称ではない
- 第三者が運営するサービスを業務に持ち込む場合、サービス側の規約とプロバイダ側の規約は同時に適用される
- プロンプトとコンテキストはセットで定義が決まるため、ペアで参照すると意味の輪郭が安定する
- トークンは課金・上限のいずれにも基準として使われ、料金と入力サイズの話題で参照される
