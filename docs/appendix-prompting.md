# Appendix: プロンプトの組み立て方

生成AIの出力品質は、モデルの性能だけでなく、依頼文（プロンプト）の組み立て方で大きく変わります。本付録は、[7章](07-terminology.md)で定義した「プロンプト＝AIへの依頼文」を出発点に、業務で使える組み立てパターンを具体例つきで整理します。

特定のモデルやサービスに依存するテクニックは扱いません。Claude・Geminiの両方で共通して使える考え方に絞ります。個別サービスの機能は[11章](11-gemini-advanced.md)・[13章](13-claude.md)を参照してください。

## 対象読者と前提

- [7章（用語集）](07-terminology.md)でプロンプト・コンテキストの定義を把握した人
- [8章（共通編）](08-common-capabilities.md)でチャット・アーティファクト・コネクタの3本柱を眺めた人
- 「もう少しうまく頼めば、もっとよい結果が返ってくるのでは」と感じている人

## 依頼文の基本形: 役割・背景・出力形式

プロンプトに盛り込む情報は、次の3要素に分けると整理しやすくなります。

| 要素 | 何を書くか | 例 |
| ---- | ---- | ---- |
| 役割 | AIにどういう立場で答えてほしいか | 「あなたは社内向けドキュメントの編集者です」 |
| 背景 | 依頼の前提や制約 | 「読者はITリテラシーのある非エンジニアです」 |
| 出力形式 | どんな形で返してほしいか | 「箇条書き5項目以内で」「表形式で」 |

3要素をすべて含む必要はありません。簡単な質問なら背景だけで十分ですし、出力形式を指定しなくても適切な形が返ることもあります。指定を増やすほど出力の揺れは小さくなるため、結果が安定しないときに要素を足していく使い方が実用的です。

### 3要素を入れた場合と入れない場合の差

同じ依頼を、要素なしと要素ありで比較してみます。

要素なしの場合。

```text
来週の定例会の議題を考えて
```

3要素を入れた場合。

```text
あなたはプロジェクトの進行管理担当です。
来週のチーム定例会（30分）の議題を考えてください。
前回はリリーススケジュールの遅延について話しました。
メンバーは5人で、開発・デザイン・QAが参加します。
箇条書き3〜5項目で、各項目に想定所要時間を添えてください。
```

前者でも回答は返りますが、汎用的な議題が並びやすく、自分の会議にそのまま使える確率は下がります。後者のように前提を渡すと、モデルの参照材料が増え、出力の的中率が上がります。

## 段階に分けて進める

1回のプロンプトに全部を詰め込むより、段階を分けて応答を確認しながら進めるほうが、結果として修正の往復が少なくなります。[8章](08-common-capabilities.md)でも触れた考え方を、もう少し具体化します。

段階分割が向くのは、次のような場面です。

- 長い文書の要約 — まず構成を把握させ、次に要約させる
- 下書きの作成 — まず骨子を出させ、合意してから本文を書かせる
- 分析と提案 — まず事実を整理させ、次に提案を求める

具体的な流れの例を示します。提案書の下書きを頼む場合です。

```text
【1回目】以下の製品仕様を読んで、要点を5つに整理してください。
（仕様のテキストを貼る）

【2回目】その5つの要点をもとに、提案書の構成案を作ってください。
見出しだけ並べてください。

【3回目】構成案の2番目の見出しについて、本文を書いてください。
読者は技術に詳しくない経営層です。
```

1回目で素材の理解を揃え、2回目で骨格を合意し、3回目で本文に進みます。いきなり「提案書を書いて」と頼むと、構成と文体の両方がモデル任せになるため、手戻りが大きくなりがちです。

## 例示パターン: 「こう返してほしい」の見本を添える

出力のフォーマットや文体を揃えたいとき、言葉で指示するよりも、見本を1〜2件プロンプトに含めるほうが安定します。

```text
以下の形式で、製品の特徴を1行にまとめてください。

見本:
- Slack: チーム内のリアルタイムなメッセージングとチャンネル整理
- Notion: ドキュメント・タスク・データベースを1つの画面にまとめる

対象:
- Figma
- Linear
- Miro
```

見本を見たモデルは、粒度・語数・構文を揃えて応答を返しやすくなります。見本の数は1〜3件で十分です。多すぎると逆にコンテキストを圧迫し、本題の処理に使える領域が減ります。

見本を使う場面の典型は次の3つです。

- 定型フォーマットの繰り返し — 日報、レビューコメント、用語説明
- 文体の統一 — 社内向け、社外向け、カジュアル、フォーマルの書き分け
- 分類やラベル付け — 問い合わせの種別分け、優先度の振り分け

## 業務シーン別のテンプレート

ここまでの3つのパターン（基本形・段階分割・例示）を組み合わせた、業務で使いやすいテンプレートを5つ示します。そのままコピーして使っても構いませんし、自分の業務に合わせて書き換えて使ってください。

### 要約

```text
以下の文章を、3つの箇条書きで要約してください。
各項目は1〜2文にしてください。
専門用語が出てきたら、括弧内に平易な言い換えを添えてください。

（本文を貼る）
```

### 翻訳・トーン調整

```text
以下のメールを英語に翻訳してください。
相手は初めてやり取りする取引先の担当者です。
丁寧だが堅すぎないビジネストーンでお願いします。

（メール本文を貼る）
```

### 下書き

```text
以下の箇条書きをもとに、社内向けの案内文を書いてください。
読者はエンジニア以外の社員です。200字程度でまとめてください。

要点:
- （要点1）
- （要点2）
- （要点3）
```

### 比較整理

```text
以下の2つの選択肢を、次の観点で比較する表を作ってください。
観点: コスト、導入の手間、既存ツールとの連携

選択肢A: （説明を貼る）
選択肢B: （説明を貼る）
```

### 壁打ち・相談

```text
新規プロジェクトの企画を検討しています。
以下の方針について、見落としている観点や懸念点を3つ挙げてください。
指摘ごとに、なぜそれが問題になりうるかを1文で添えてください。

方針: （方針の説明を貼る）
```

テンプレートは出発点です。最初の応答を見て、足りない情報を補ったり出力形式を調整したりする往復が、プロンプトの精度を上げていく通常の流れにあたります。

## やりがちな失敗パターン

- 一度に全部を頼む — 素材の理解、構成、本文、校正を1回のプロンプトで求めると、どの段階の判断もモデル任せになる。段階を分ける
- 背景を省いて形式だけ指定する —「表にして」だけでは、何を軸に整理すべきか分からず、汎用的な表が返ってくる。比較の観点や読者の前提を添える
- 長大な資料をそのまま貼る — コンテキストウィンドウ（[7章](07-terminology.md)）の上限を超えた部分は参照されない。要点を抜くか、段階に分けて渡す
- 見本なしにフォーマットを期待する —「いい感じに」では揺れが出る。1件でよいので見本を置く
- 手戻りを恐れて1回で決めようとする — 最初の応答がそのまま完成品になることは多くない。応答を見て追加の指示を出す往復が、プロンプト活用の通常の進め方にあたる

## まとめ

- プロンプトの基本形は、役割・背景・出力形式の3要素で整理できる。結果が安定しないときに要素を足していく使い方が実用的
- 1回で全部を頼むより、段階を分けて応答を確認しながら進めるほうが修正の往復は少なくなる
- 出力のフォーマットや文体を揃えたいときは、見本を1〜2件プロンプト内に含めると安定する
- テンプレートは出発点であり、応答を見て情報を補う往復がプロンプト活用の通常の流れにあたる
