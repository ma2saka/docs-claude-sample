# Appendix: プロンプトの組み立て方

生成AIの出力品質は、モデルの性能だけでなく、依頼文（プロンプト）の組み立て方で大きく変わります。本付録は、[7章](07-terminology.md)で定義した「プロンプト＝AIへの依頼文」を出発点に、業務で使える組み立てパターンを具体例つきで整理します。

特定のモデルやサービスに依存するテクニックは扱いません。Claude・Geminiの両方で共通して使える考え方に絞ります。テキスト入力を中心に扱いますが、音声入力での依頼パターンにも1節を割いています。個別サービスの機能は[11章](11-gemini-advanced.md)・[13章](13-claude.md)を参照してください。

## 対象読者と前提

- [7章（用語集）](07-terminology.md)でプロンプト・コンテキストの定義を把握した人
- [8章（共通編）](08-common-capabilities.md)でチャット・アーティファクト・コネクタの3本柱を眺めた人
- [9章（セキュリティ 個人利用編）](09-security-individual.md)で、入力データの判断基準に目を通した人
- 「もう少しうまく頼めば、もっとよい結果が返ってくるのでは」と感じている人

## 依頼文の基本形: 役割・背景・出力形式

プロンプトに盛り込む情報は、役割・背景・出力形式の3要素で整理できます。

| 要素 | 何を書くか | 例 |
| ---- | ---- | ---- |
| 役割 | AIにどういう立場で答えてほしいか | 「あなたは社内向けドキュメントの編集者です」 |
| 背景 | 依頼の前提や制約 | 「読者はITリテラシーのある非エンジニアです」 |
| 出力形式 | どんな形で返してほしいか | 「箇条書き5項目以内で」「表形式で」 |

3要素をすべて含む必要はありません。簡単な質問なら背景だけで十分ですし、出力形式を指定しなくても適切な形が返ることもあります。指定を増やすほど出力の揺れは小さくなるため、結果が安定しないときに要素を足していく使い方が実用的です。

### 3要素を入れた場合と入れない場合の差

同じ依頼を、要素なしと要素ありで比較してみます。

まず要素なしで書いた場合です。

```text
来週の定例会の議題を考えて
```

次に3要素を入れた場合です。

```text
あなたはプロジェクトの進行管理担当です。
来週のチーム定例会（30分）の議題を考えてください。
前回はリリーススケジュールの遅延について話しました。
メンバーは5人で、開発・デザイン・QAが参加します。
箇条書き3〜5項目で、各項目に想定所要時間を添えてください。
```

前者でも回答は返りますが、汎用的な議題が並びやすく、自分の会議にそのまま使える確率は下がります。後者のように前提を渡すと、モデルの参照材料が増え、出力の的中率が上がります。

## 段階に分けて進める

1回のプロンプトに全部を詰め込むより、段階を分けて応答を確認しながら進めるほうが、結果として修正の往復が少なくなります。[8章](08-common-capabilities.md)でも触れた考え方を、もう少し具体化します。

段階分割が向くのは、次のような場面です。

- 長い文書の要約 — まず構成を把握させ、次に要約させる
- 下書きの作成 — まず骨子を出させ、合意してから本文を書かせる
- 分析と提案 — まず事実を整理させ、次に提案を求める

具体的な流れの例を示します。提案書の下書きを頼む場合です。

```text
【1回目】以下の製品仕様を読んで、要点を5つに整理してください。
（仕様のテキストを貼る）

【2回目】その5つの要点をもとに、提案書の構成案を作ってください。
見出しだけ並べてください。

【3回目】構成案の2番目の見出しについて、本文を書いてください。
読者は技術に詳しくない経営層です。
```

1回目で素材の理解を揃え、2回目で骨格を合意し、3回目で本文に進みます。いきなり「提案書を書いて」と頼むと、構成と文体の両方がモデル任せになるため、手戻りが大きくなりがちです。

## 例示パターン: 「こう返してほしい」の見本を添える

出力のフォーマットや文体を揃えたいとき、言葉で指示するよりも、見本を1〜2件プロンプトに含めるほうが安定します。

```text
以下の形式で、製品の特徴を1行にまとめてください。

見本:
- Slack: チーム内のリアルタイムなメッセージングとチャンネル整理
- Notion: ドキュメント・タスク・データベースを1つの画面にまとめる

対象:
- Figma
- Linear
- Miro
```

見本を見たモデルは、粒度・語数・構文を揃えて応答を返しやすくなります。見本の数は1〜3件で十分です。多すぎると逆にコンテキストを圧迫し、本題の処理に使える領域が減ります。

見本を使う場面の典型は次の3つです。

- 定型フォーマットの繰り返し — 日報、レビューコメント、用語説明
- 文体の統一 — 社内向け、社外向け、カジュアル、フォーマルの書き分け
- 分類やラベル付け — 問い合わせの種別分け、優先度の振り分け

## 業務シーン別のテンプレート

ここまでの3つのパターン（基本形・段階分割・例示）を組み合わせた、業務で使いやすいテンプレートを9つ示します。そのままコピーして使っても構いませんし、自分の業務に合わせて書き換えて使ってください。

### 要約

```text
以下の文章を、3つの箇条書きで要約してください。
各項目は1〜2文にしてください。
専門用語が出てきたら、括弧内に平易な言い換えを添えてください。

（本文を貼る）
```

### 翻訳・トーン調整

```text
以下のメールを英語に翻訳してください。
相手は初めてやり取りする取引先の担当者です。
丁寧だが堅すぎないビジネストーンでお願いします。

（メール本文を貼る）
```

### 下書き

```text
以下の箇条書きをもとに、社内向けの案内文を書いてください。
読者はエンジニア以外の社員です。200字程度でまとめてください。

要点:
- （要点1）
- （要点2）
- （要点3）
```

### 比較整理

```text
以下の2つの選択肢を、次の観点で比較する表を作ってください。
観点: コスト、導入の手間、既存ツールとの連携

選択肢A: （説明を貼る）
選択肢B: （説明を貼る）
```

### 壁打ち・相談

```text
新規プロジェクトの企画を検討しています。
以下の方針について、見落としている観点や懸念点を3つ挙げてください。
指摘ごとに、なぜそれが問題になりうるかを1文で添えてください。

方針: （方針の説明を貼る）
```

### 議事録の整理

```text
以下は会議中に取ったメモ（または自動文字起こし）です。
参加者: （名前を列挙）
会議の目的: （目的を1文で）

次の構成で議事録にまとめてください。
1. 日時・参加者
2. 議題ごとの要点（発言者がわかる場合は名前つき）
3. 決定事項
4. 次のアクション（担当と期限つき）

（メモまたは文字起こしを貼る）
```

会議メモや文字起こしには、脱線・重複・話の前後が混在します。上記のように出力の構成を先に示すと、モデルが情報を再配置する方針を持てるため、整理の精度が上がります。Meetの文字起こしを素材にする場合は、[12章](12-google-workspace-and-gemini.md)のMeet節も参照してください。

### データの読み解きと整形

```text
以下のデータを、指定の形式に整理してください。
データの種類: （例: 月別の売上、アンケートの自由記述、問い合わせログ）
整理の目的: （例: 月次報告の材料にする、傾向を把握する）
出力形式: （例: カテゴリ別の集計表、上位5件の一覧、傾向の箇条書き3点）

（データを貼る）
```

数字やテキストの羅列をそのまま渡すだけでは、モデルはどの軸で切るべきか判断できません。「何のために整理するか」と「どんな形で返してほしいか」の2点を添えると、出力のずれが小さくなります。

### メール返信の下書き

```text
以下の受信メールに対する返信を書いてください。
返信の方針: （例: 依頼を承諾する、日程の再調整を提案する、追加情報を求める）
トーン: （例: 社内向けのカジュアル、社外向けの丁寧語）
補足したい情報: （例: 「納期は来週金曜に変更」「代替案として○○を提示」）

受信メール:
（受信メールを貼る）
```

翻訳テンプレートとの違いは、入力が「自分の文章」ではなく「相手の文章＋返信方針」である点です。返信方針を明示しないと、モデルは受信メールの内容を要約するだけの応答を返しやすくなります。

### レビュー・チェック依頼

```text
以下の文書を、次の観点でレビューしてください。
目的: （例: 社外に送る前の最終チェック、上長への提出前の品質確認）
確認してほしい観点:
- 論理の一貫性（主張と根拠がつながっているか）
- 事実関係の疑わしい箇所
- 抜けている観点や情報
- 読者にとってわかりにくい表現

指摘は箇条書きで、該当箇所の引用と修正案を添えてください。
問題が見当たらなければ「確認した範囲で指摘事項はありません」と返してください。

（文書を貼る）
```

ここまでのテンプレートが「新しく作る」「形を変える」用途であるのに対し、レビュー依頼は「できあがったものを検証する」用途にあたります。確認してほしい観点を列挙して渡すと、AIの指摘が特定の方向に偏りにくくなります。観点のリストは業務に合わせて差し替えてください。たとえば、社外向け提案書なら「競合との差別化が伝わるか」、社内手順書なら「前提知識のない人が読んで手順どおりに進められるか」を加えると、チェックの焦点が定まります。

AIのレビューは、人が目を通す前の下準備として位置づけます。事実関係の最終確認は[6章](06-hallucination-and-knowledge-literacy.md)で整理した裏取りの手順に沿って、利用者が行います。

テンプレートは出発点です。最初の応答を見て、足りない情報を補ったり出力形式を調整したりする往復が、プロンプトの精度を上げていく通常の流れにあたります。

## カスタム指示: 毎回書き直さなくてよい前提をまとめておく

ここまでのテンプレートは、会話のたびに入力するプロンプトです。同じ前提を繰り返し使う場合は、その前提を**カスタム指示**としてあらかじめ登録しておくと、毎回書き直す手間がなくなります。

GeminiではGems、ClaudeではProjectsのカスタム指示欄がこの仕組みにあたります。セットアップの手順はそれぞれ[11章（Gems）](11-gemini-advanced.md)・[13章（Projects）](13-claude.md)で扱っているため、本節ではカスタム指示に**何を書くか**の組み立て方に集中します。

### 会話用プロンプトとの違い

会話用のプロンプトは「今回のタスク」に合わせて書きます。一方、カスタム指示はそのGemやProject内で行う**すべての会話に共通する前提**を書く場所です。違いを表で示します。

| 観点 | 会話用プロンプト | カスタム指示 |
| ---- | ---- | ---- |
| 有効範囲 | その1回の会話だけ | GemまたはProject内のすべての会話 |
| 書く内容 | 具体的なタスクと素材 | 役割・読者像・口調・出力形式の既定値 |
| 変更の頻度 | 毎回書き直す | 一度書いたら、月に1回程度の見直しで済む |

カスタム指示は「このGem（またはProject）の中では、いつもこの前提で話してほしい」という宣言です。個別のタスクに依存する情報（素材のテキスト、今回の依頼内容など）はカスタム指示には含めず、会話時のプロンプトに書きます。

### カスタム指示に書くこと・書かないこと

盛り込む内容は、本付録の冒頭で紹介した「役割・背景・出力形式」の3要素から、**タスクに依存しない部分だけ**を抜き出す形です。書くとよいものと書かないほうがよいものを並べます。

| 分類 | 内容 | 例 |
| ---- | ---- | ---- |
| 書く | 役割の固定 | 「社内向けドキュメントの編集者として振る舞う」 |
| 書く | 読者像の固定 | 「読者はITリテラシーのある非エンジニアである」 |
| 書く | 口調・文体 | 「です・ます調で、1文を短く保つ」 |
| 書く | 出力形式の既定値 | 「表で比較を求められたら、列名を必ず日本語にする」 |
| 書く | 制約 | 「回答はナレッジに登録した資料の範囲内に限定する」 |
| 書かない | 具体的なタスク指示 | 「この議事録を要約して」のような1回限りの依頼は会話時に書く |
| 書かない | 素材のテキスト | 参照資料はナレッジやファイルアップロードに分離する |
| 書かない | 網羅的なルールの羅列 | 項目が多すぎるとコンテキストを圧迫する。10項目を目安に絞る |

### カスタム指示のテンプレート3例

業務でそのまま使える例を3つ示します。

#### 週次レポート補助

```text
あなたは社内向け週次レポートの編集補助者です。

前提:
- 読者は同じ部門の上長とメンバー（技術の深い知識は前提にしない）
- レポートの構成は「今週の成果」「来週の予定」「課題・相談事項」の3節

出力ルール:
- です・ます調
- 各節は箇条書き3〜5項目
- 数字（件数・日付・金額）は省略せずそのまま残す
- 課題の項目には、想定される対応策を1文添える
```

#### 英文メール推敲

```text
あなたは英文ビジネスメールの推敲者です。

前提:
- 差出人は日本企業の担当者で、相手は海外の取引先である
- メールの目的は依頼文を入力するので、そこから判断する

出力ルール:
- 原文の意図を変えずに、文法・語法・トーンを修正する
- 修正した箇所を太字で示す
- 修正理由を箇条書きで末尾にまとめる
- 丁寧だが堅すぎないビジネストーンを基準にする
```

#### 社内FAQ回答

```text
あなたは社内ヘルプデスクの回答担当者です。

前提:
- 質問者は社内の非エンジニア
- 回答に使ってよい情報はナレッジに登録された社内マニュアルの範囲内のみ

出力ルール:
- 回答は3〜5文以内に収める
- 手順がある場合は番号付きリストで示す
- ナレッジ内に該当する情報がない場合は「該当する情報が見つかりません。○○チームへの問い合わせをお勧めします」と返す
- 推測で補わない
```

### カスタム指示を育てるサイクル

カスタム指示は、最初から完成形で書く必要はありません。次の手順で段階的に整えるのが実用的です。

1. 本付録の業務テンプレートの中から、繰り返し使っているものを1つ選ぶ
2. そのテンプレートから、タスク固有の部分（素材のテキスト、具体的な依頼文）を抜き、残った前提部分をカスタム指示に登録する
3. 数回の会話で試し、応答の傾向を見ながら指示を調整する。口調が安定しなければ文体の項目を足し、出力が長すぎれば分量の制約を足す

カスタム指示の設定場所と手順は、Geminiは[11章のGems節](11-gemini-advanced.md)、Claudeは[13章のProjects節](13-claude.md)を参照してください。

## プロンプトに渡す情報の判断基準

プロンプトの品質を上げるために背景や素材を充実させるのは、ここまで見てきたとおりです。ただし、業務で扱う情報の中には、そのままの形では渡すべきでないものが含まれます。プロンプトに情報を貼り付けることは、その情報をサービスの通信経路に乗せることを意味します。

判断の基本は[9章（セキュリティ 個人利用編）](09-security-individual.md)で整理した3つの観点です。

| 観点 | プロンプト作成時に確認すること |
| ---- | ---- |
| 機密度 | 顧客名・契約金額・未公開情報・個人情報・認証情報が含まれていないか |
| 第三者権利 | 他社から受領したNDA付き資料や、他者の著作物をそのまま貼っていないか |
| 法規制 | マイナンバー・医療情報・決済情報など、法令上の取扱制限がある情報を含んでいないか |

3つのうち1つでも該当する場合は、渡す前にひと手間を加えます。判断に迷う場合の進め方は[9章の判断フロー](09-security-individual.md)を参照してください。

### 機密情報を含む業務でもAIを活用する3つのパターン

機密情報に触れる業務だからAIを使えない、とは限りません。渡す情報の形を変えることで、AIの出力を業務に活用する余地は残ります。

#### パターン1: 固有名詞を仮名に置き換える

顧客名や社名をA社・B社に、担当者名を「担当者X」に置き換えてからプロンプトに渡します。応答を受け取ったあとに、手元で元の名称に差し戻します。

```text
以下の商談メモをもとに、次回の提案方針を箇条書きで整理してください。

- A社（製造業、従業員500名規模）
- 課題: 在庫管理の属人化
- 予算感: 年間1,000万円前後
- 競合: B社のクラウドサービスを検討中
```

#### パターン2: 構造だけを渡し、具体値は手元で埋める

AIに考えてほしいのが構成や文面の骨格であれば、具体的な数字や名称は伏せたまま構造を渡すだけで十分です。

```text
以下の項目構成で提案書のアウトラインを作成してください。
各項目に1〜2文の説明文をつけてください。

1. 現状の課題（3点）
2. 提案するソリューションの概要
3. 導入スケジュール（3か月想定）
4. 概算費用
5. 期待される効果
```

具体的な費用や社名は、受け取ったアウトラインに手元で書き込みます。

#### パターン3: 公開情報だけを組み合わせる

社内の機密データではなく、公開されている業界情報・統計・記事だけを材料にして依頼を組み立てます。Web検索が有効な場合は、URLを添えるとモデルが参照元を辿りやすくなります。

3つのパターンに共通するのは、AIには「考える仕事」を渡し、「機密の具体値を扱う仕事」は手元に残す、という切り分けです。この切り分けは、機密が外へ流れる経路を狭めるだけでなく、余分な情報が減る分だけモデルが依頼の主旨を拾いやすくなるため、応答の品質にも作用します。

組織としてAIの利用ガイドラインが定められている場合は、そちらが優先されます。ガイドラインの読み解き方や、自分の利用がどの経路に該当するかの判断は[9章](09-security-individual.md)で扱っています。

## やりがちな失敗パターン

- 一度に全部を頼む — 素材の理解、構成、本文、校正を1回のプロンプトで求めると、どの段階の判断もモデル任せになる。段階を分ける
- 背景を省いて形式だけ指定する —「表にして」だけでは、何を軸に整理すべきか分からず、汎用的な表が返ってくる。比較の観点や読者の前提を添える
- 長大な資料をそのまま貼る — コンテキストウィンドウ（[7章](07-terminology.md)）の上限を超えた部分は参照されない。要点を抜くか、段階に分けて渡す
- 見本なしにフォーマットを期待する —「いい感じに」では揺れが出る。1件でよいので見本を置く
- 機密情報をそのまま貼り付ける — 顧客名や契約金額を含む素材をそのまま渡している。固有名詞の仮名化や構造だけの抽出で、情報の形を変えてから渡す（前節の3パターンを参照）
- 手戻りを恐れて1回で決めようとする — 最初の応答がそのまま完成品になることは多くない。応答を見て追加の指示を出す往復が、プロンプト活用の通常の進め方にあたる

## 応答を立て直すときの定型パターン

前節の失敗パターンで触れたとおり、応答を見て追加の指示を出す往復がプロンプト活用の通常の進め方です。ここで課題になるのは、応答が期待とずれたときに次の修正指示をどう組み立てるかです。本節では、ずれの類型ごとに使える修正指示のパターンを6つ示します。

### 長さが合わないとき

応答が長すぎる、または短すぎる場合です。分量の指定がないと、モデルは依頼の曖昧さに応じて長めに書く傾向があります。

```text
300字以内に縮めてください。削る優先順位は、具体例 → 補足説明 → 結論の順です。
```

```text
各項目にもう1〜2文ずつ補足を足してください。特に「導入の手間」の項目が薄いです。
```

分量の目安を数字で渡すのが安定します。「もう少し短く」のような相対指定は、モデルによって解釈の幅が出やすいため、具体的な字数や文数で指定するほうが修正1回で着地します。

### 焦点がずれているとき

依頼の意図と異なる方向に応答が逸れている場合です。モデルは依頼文の中で目立つキーワードに引っ張られて、主題から外れた方向へ展開することがあります。

```text
質問の焦点は「社内展開のリスク」です。技術的な仕組みの説明は不要なので省いてください。
リスクの種類を3つに絞り、それぞれ対応策を1文で添えてください。
```

ずれを指摘するだけでなく「不要な方向」と「戻ってほしい方向」の両方を書くと、修正の精度が上がります。「違います」だけでは、モデルは別の方向へずれ直すことがあります。

### 粒度が合わないとき

抽象的すぎて実務に落とし込めない、あるいは細かすぎて全体像が見えない場合です。

```text
「コスト削減につながる」だけでは判断材料になりません。
どの工程で、どのくらいの時間短縮が見込めるか、具体的な数字の目安を入れてください。
```

```text
個々のAPI仕様まで踏み込みすぎています。
「利用者が設定画面で操作できる範囲」の粒度でまとめ直してください。
```

「もっと具体的に」「もっと概要で」という指示は曖昧になりがちです。どの項目がどう粗い（あるいは細かい）のかを名指しし、求める粒度を1つの例で示すと、意図が伝わりやすくなります。

### 出力形式を変えたいとき

箇条書きで返ってきたものの表のほうが読みやすい、段落で返ってきたものの箇条書きのほうが共有しやすい、といった場合です。

```text
この内容を表にしてください。列は「項目名」「メリット」「注意点」の3列です。
```

```text
表ではなく、各選択肢を見出し付きの段落で書き直してください。
各段落の冒頭に、その選択肢を一文で要約してください。
```

形式の変更は、列名や見出し構成など出力の枠組みを指定すると安定します。先に紹介した例示パターン（見本を1件添える方法）と組み合わせると、形式と粒度の両方を1回の往復で揃えられます。

### 事実が怪しいとき

応答に含まれる情報が正しいか確信が持てない場合です。[6章](06-hallucination-and-knowledge-literacy.md)で整理したとおり、生成AIは自然な体裁のまま事実誤認を混ぜることがあります（ハルシネーション）。修正指示だけで問題が解消するとは限らないため、一次ソースでの裏取りと組み合わせる前提で使います。

```text
3番目の項目に書かれている数字の根拠を示してください。
公式ドキュメントや発表資料のURLがあれば添えてください。
```

```text
この回答に含まれる事実関係のうち、確信度が低いものがあれば明示してください。
```

根拠の提示を求める指示は、モデルに自己点検を促す一歩目として有効です。ただし、モデルの示すURL自体が存在しないこともあるため、最終的な検証は利用者が一次ソースを開いて行う必要があります。裏取りの段取りは[6章の「出力の裏取りは利用者の責任である」節](06-hallucination-and-knowledge-literacy.md)で扱っています。

### トーンが合わないとき

文体が堅すぎる、くだけすぎる、あるいは特定の読者層に合っていない場合です。

```text
社外の取引先に送るメールとしては砕けすぎています。
敬語を基調にしつつ、箇条書き部分は簡潔に保ってください。
以下のトーンを参考にしてください。

参考: 「本件について、3点ご確認をお願いいたします。」
```

```text
社内のSlack投稿としては堅すぎます。
「です・ます」は残しつつ、1文を短くして読みやすくしてください。
```

トーンの指示は主観的になりやすいため、見本を1文添えるのが効果的です。「もう少しカジュアルに」のような抽象指示より、具体的な文例のほうがモデルの解釈の揺れが小さくなります。

### 立て直しの手順をまとめると

6つのパターンに共通する手順は次の3段階です。

1. **ずれの特定** — 応答のどの部分が、どう期待と違うかを言語化する
2. **修正指示の送信** — ずれた箇所を名指しし、求める方向を具体的に書く。数字・見本・列名など、解釈の揺れが出にくい形式で指定する
3. **再評価** — 修正後の応答を確認し、必要なら再度修正する。事実関係の確認は一次ソースへの裏取りを別途行う

2〜3回の往復で着地しない場合は、プロンプトの前提（役割・背景）を見直すか、依頼を分割して段階に分けるほうが近道です。同じ会話の流れを続けて修正することにこだわるより、新しい会話で仕切り直すと、蓄積した履歴にモデルが引きずられる問題を避けられます。

## 音声で依頼するときの組み立て方

Gemini Live（[11章](11-gemini-advanced.md)）をはじめ、音声で生成AIに依頼できる入口が増えています。本付録がここまで扱ってきたパターン（役割・背景・出力形式の3要素、段階分割、立て直し）はテキスト入力を前提にしていますが、音声で依頼する場面でも同じ考え方の骨格は使えます。ただし、入力の性質が変わるため、組み立て方にいくつかの調整が要ります。

### テキストと音声で変わること

| 観点 | テキスト入力 | 音声入力 |
| ---- | ---- | ---- |
| 1回の依頼に詰め込める情報量 | 長文を構造的に書ける | 短い発話を重ねるほうが安定する |
| 出力形式の指定 | 列名や見出し構成まで細かく指定できる | 「箇条書きで」「3つに絞って」程度の粗い指定が実用的 |
| 材料の渡し方 | テキストの貼り付け、ファイルの添付 | 画面共有やタブ共有で材料を渡し、口頭では概要だけ伝える |
| 修正指示 | 該当箇所を引用しながら指定できる | 「2番目の項目をもう少し具体的に」のように位置と方向を口頭で示す |
| 記録の残り方 | そのままコピー＆ペーストで再利用できる | テキスト化は自動で行われるが、構造の正確さはテキスト入力に劣る |

要約すると、テキスト入力は「1回で正確に伝える」ことに向き、音声入力は「短いやり取りを重ねて方向を絞っていく」ことに向きます。

### 音声入力で実用的な3つのパターン

#### 壁打ち — 方向性を口頭で固める

まずテーマと論点を口頭で伝え、AIの応答を受けて掘り下げます。最初から完成形を求めず、「この方向で合っているか確認したい」「別の切り口があれば挙げてほしい」のように、判断材料を引き出す使い方です。

```text
来週の提案書の方向性を相談したい。
顧客は製造業で、課題は在庫管理の属人化。
提案の軸として考えているのはデータの可視化だけど、
ほかに有効な切り口があれば3つ挙げてほしい。
```

#### タブや画面を材料にした要約・質問

Gemini in Chromeのサイドパネルや、モバイルアプリのカメラ入力など、目の前の情報を材料にして音声で質問します。材料はすでに画面上にあるため、口頭では「何をしてほしいか」だけを短く伝えます。

```text
このページの要点を3つ教えて。
```

```text
この2つのタブの料金プランを比較して、違いだけ教えて。
```

#### テキストへの切り替え — 方針確定後に成果物を仕上げる

音声で方針を固めたあと、成果物の作成はテキストチャットに移ります。音声で得た結論を冒頭に書き、出力形式を指定してテキストで仕上げるのが、品質と再利用性の両面で安定する流れです。

### 音声とテキストの使い分けの目安

| 段階 | 向いている入力 | 理由 |
| ---- | ---- | ---- |
| テーマの探索・壁打ち | 音声 | 考えながら話すほうが自然。方向の修正も口頭のやり取りで進めやすい |
| 論点の洗い出し・比較 | 音声 | 材料を見ながら「これとこれの違いは」と口頭で掘り下げられる |
| 成果物の作成・推敲 | テキスト | 出力形式を細かく指定でき、結果をそのまま転記・共有できる |
| 複雑な条件の指定 | テキスト | 条件が3つ以上あるとき、口頭では抜け落ちや曖昧さが出やすい |

音声対話の機能詳細（利用できる入口・プランの条件・テキストチャットとの違い）は[11章のGemini Live節](11-gemini-advanced.md)を参照してください。

## 長い作業を複数の会話に分けて進める

ここまでの節は、1回の会話の中で依頼を組み立て、応答を修正する方法を扱ってきました。しかし業務では、調査・構成・執筆・推敲のように段階の異なる作業をまたぐ場面や、数日にわたって1つのテーマに取り組む場面があります。こうした作業を1つの会話で最後まで通そうとすると、途中から応答の質が落ちたり、前半に伝えた前提が反映されなくなったりする現象が起きます。

本節では、この現象の背景と、会話を分けて進めるための実践パターンを示します。

### 会話が長くなると起きること

[7章（用語集）](07-terminology.md)で定義したとおり、AIが1回の応答を生成する際に参照できる情報の総量には上限があります。この上限がコンテキストウィンドウです。会話が長くなると、過去のやり取りがコンテキストウィンドウの中で占める割合が増え、古いやり取りから順に参照されにくくなります。

利用者の側から見ると、次のような形で現れます。

- 会話の前半で共有した前提条件が、後半の応答に反映されなくなる
- 同じ質問に対して、会話の序盤とは異なる方向の回答が返る
- 修正指示を重ねるほど、応答が以前の修正と矛盾する方向へぶれる

これらは、モデルの能力が途中で下がったのではなく、参照できる情報の範囲が押し出されたことによる現象です。「AIが疲れた」のではなく、机の上の資料が積み上がって古い資料が下に埋もれた状態にあたります（[7章](07-terminology.md)の机の比喩を参照してください）。

### 会話を切り替える目安

明確な閾値はありませんが、次の兆候が出たときは、新しい会話に切り替えるタイミングです。

- 前半で伝えた制約や条件が応答から抜け落ちている
- 話題が当初のテーマから大きく変わった（調査の途中で別の企画の相談に移った、など）
- 立て直しの修正指示を3回以上繰り返しても着地しない
- 成果物（要約・表・下書き）が一段落し、次の段階へ進む区切りにある

会話を切り替えること自体にコストはありません。むしろ、切り替えたほうが次の作業に集中した応答が得られます。

### 成果を次の会話に引き継ぐ3つのパターン

新しい会話を始めると、前の会話のやり取りは参照されなくなります。前の会話で得た成果を次に持ち込むには、次の3つのパターンがあります。

| パターン | やり方 | 向いている場面 |
| ---- | ---- | ---- |
| 成果物の貼り付け | 前の会話で得た要約・表・箇条書きを、次の会話の冒頭に貼る | 段階ごとに作業を区切って進めるとき |
| カスタム指示への登録 | 繰り返す前提をGems（Gemini）やProjects（Claude）のカスタム指示に書いておく | 同じテーマで何度も会話するとき |
| メモリ・保存済み情報の活用 | 利用者の情報や好みをメモリ（[13章](13-claude.md)）や保存済み情報（[11章](11-gemini-advanced.md)）に蓄積する | 利用者自身の立場や好みを毎回伝え直したくないとき |

もっとも手軽で汎用性が高いのは1番目の「成果物の貼り付け」です。前の会話の最後に「ここまでの要点を箇条書き5項目でまとめてください」と依頼し、その出力を次の会話の冒頭に添えます。会話を閉じる前にまとめを作る習慣をつけると、引き継ぎが途切れません。

2番目のカスタム指示は、本付録の「カスタム指示」節で扱った仕組みの応用です。プロジェクト単位で口調や制約が決まっている場合、それをカスタム指示に登録しておくと、新しい会話でも同じ前提が自動的に適用されます。

3番目のメモリ・保存済み情報は、利用者個人に紐づく長期的な記憶です。名前・職種・文体の好みなど、プロジェクトをまたいでも変わらない情報の保持に向きます。設定手順は[13章（Claudeのメモリ）](13-claude.md)・[11章（Geminiの保存済み情報）](11-gemini-advanced.md)を参照してください。

### 業務での典型的な分割例: 企画書の作成

1つの作業を4つの会話に分けて進める例を示します。

| 会話 | 目的 | 冒頭に渡すもの | 会話の終わりに作るもの |
| ---- | ---- | ---- | ---- |
| 1回目 | 調査と情報収集 | テーマと調査の観点 | 調査結果の要約（箇条書き） |
| 2回目 | 構成の設計 | 1回目の要約 + 企画書の目的と読者 | 章立てと各章の要点（箇条書き） |
| 3回目 | 本文の執筆 | 2回目の章立て + 補足資料 | 本文の下書き |
| 4回目 | 推敲と仕上げ | 3回目の下書き + 修正方針 | 完成版 |

各会話は短く保たれるため、コンテキストの圧迫が起きにくく、段階ごとに集中した応答が得られます。1回目で調査を依頼し、3回目でいきなり執筆へ入るといった飛ばし方も可能です。会話の合間で利用者自身が手を入れる工程を挟むと、AIの出力と利用者の判断が交互に積み重なり、成果物の品質が安定します。

カスタム指示（Gems・Projects）に企画書の口調や読者像を登録しておけば、4回の会話すべてで同じ前提が自動的に適用されます。

## まとめ

- プロンプトの基本形は、役割・背景・出力形式の3要素で整理できる。結果が安定しないときに要素を足していく使い方が実用的。音声入力では短いやり取りを重ねて方向を絞り、成果物の作成はテキストに切り替える
- 作業は段階に分けて進める。1回の会話の中では応答を確認しながら段階を踏み、1回の会話に収まらない作業は複数の会話に分けて前の成果物を次の冒頭に引き継ぐ
- 出力を安定させるには見本を含めるか、繰り返す前提をカスタム指示（Gems・Projects）にまとめる。応答がずれたときは、ずれの特定→修正指示→再評価の3段階で立て直し、事実関係の検証は一次ソースへの裏取りと組み合わせる
- 機密情報を含む業務では、AIに「考える仕事」を渡し「機密の具体値を扱う仕事」は手元に残す切り分けが基本になる。入力データの判断基準は[9章](09-security-individual.md)を参照
