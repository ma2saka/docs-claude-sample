# 章の執筆・改稿スキル

本リポジトリの章は、WRITING_GUIDE.md の「執筆の工程」に従い、構造の決定と文章化を分けて作る。一発の長文生成は、構造と文章の判断が同時に走って同型の反復に流れるため、使わない。このスキルは工程の分離を実行手順に落としたもの。

## 事前に読むもの

1. WRITING_GUIDE.md の「目指す文章」「執筆の工程」「量の目安」
2. 対象章（改稿の場合）と、前後の章（参照の整合を見るため）
3. 声のアンカーとして既存章を1〜2本（迷ったら2章と6章）

## 工程1: チャプタープラン（主エージェントが行う）

構造の判断は委譲しない。次の形式でプランを作り、作業ログ（会話またはIssueコメント）に残す。

```markdown
### 章プラン: <章タイトル>
- 解く疑問: <読者の疑問を1〜3個>
- 扱わないこと: <箇条書き>
- 節構成:
  | 節タイトル（=結論） | 主張1文 | 当てる材料 | 形式 | 軽重 |
  | --- | --- | --- | --- | --- |
- 用語の初出: <語と定義位置>
- 前後章への参照: <どの節からどこへ>
- 声のアンカー: <既存章を1〜2本>
```

プラン段階のチェック。

- [ ] 各節の主張は、節タイトルだけで結論が伝わる形か
- [ ] 同じ形式・同じ軽重の節が3つ以上並んでいないか（並んでいたらここで崩す）
- [ ] 根拠を当てられない主張が残っていないか（残っていたら主張ごと削る）

改稿の場合は逆向きに始める。現行章から各節の主張を1文ずつ抜き出してプランを復元し、主張が立たない節・同型の節をプラン側で直してから、次の工程へ進む。

## 工程2: 文章化（サブエージェントに委譲する）

Agentツールでサブエージェント（`model: "sonnet"` が目安）に委譲する。プロンプトには次を全部含め、サブエージェントが判断に迷わない状態にする。

- 工程1のプラン全文
- WRITING_GUIDE.md の「目指す文章」「文体ルール」「太字の運用」「量の目安」を読むこと
- 声のアンカー章を読み、文をそろえること
- 守る規律: プランの主張だけを書く／形式と軽重はプランに従う／節の締めの型を章内で繰り返さない／隣の節と同じ組み立てになりそうなら書かずに報告する
- 書きながら思いついた主張は本文に足さず、提案として報告に分けること

章が長い場合は節のまとまりごとに分けて依頼してよい。その際も各サブエージェントにプラン全文を渡す（自分の担当節だけ渡すと、章全体の強弱が見えず同型に流れる）。

## 工程3: 通読と推敲（主エージェントに戻す）

1. 章全体を読者として頭から読む。一次評価は2問だけ
   - 説明は前進しているか（各節は前の節が言っていないことを言っているか）
   - 流れは自然か（読者が推測で補わされる箇所はないか）
2. 機械検査を回す

   ```bash
   npm install && npm run lint && npm run style:stats -- docs/<対象>.md
   ```

3. 直すときは段落または節の単位で書き直す。語句の差し替えは置換語の連鎖を生むため行わない
4. 検査と通読で見つけた新しい反復語・定型句は、`prh-style.yml` への追加候補として報告に含める（追加自体は別PR）

## 仕上げ

- 表現のリライトでは「最終確認」日付を動かさない。事実（料金・仕様・機能名）を書き換えたときだけ、根拠URLを開き直して当該エントリの日付を更新する（詳細は source-patrol スキル）
- PRは「1章に対する加筆」を基本サイズとし、本文に「変更の要点」「確認方法」と、工程1のプラン（または改稿前後の構造の対比）を書く
- セルフレビューは工程3をもう一周する形で行い、修正がなくなるまで繰り返す
