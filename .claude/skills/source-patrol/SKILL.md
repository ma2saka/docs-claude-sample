# 参考URL・鮮度の巡回スキル

本リポジトリ（awesome docs ai works）の各章は、WRITING_GUIDE.md の「情報の鮮度」ルールに従い、参考URLごとに「最終確認： YYYY-MM-DD」を持つ。生成AI界隈は月単位で仕様が変わるため、半年〜四半期に一度の棚卸しPRと、章を触るついでの個別更新が運用の前提になっている。このスキルは、その巡回を再現可能な手順に落としたもの。

## 必ず守るルール（WRITING_GUIDE.md より）

- **URLを実際に開いて確認した日だけを「最終確認」に書く。** 開いていないのに日付だけ今日へ差し替えるのは禁止
- 章ごとに日付が混在するのは正常。揃えること自体は目的ではない
- 本文の事実（料金・仕様・機能名・モデル名）を書き換えたときは、根拠URLを開き直して当該エントリの日付を更新する
- タイポ修正など参考URLの妥当性に影響しない編集では日付を動かさない

## 手順

### 1. 対象の洗い出し

```bash
grep -n '最終確認：' docs/*.md | sed 's/（最終確認：\(.*\)）/ → \1/'
```

- 章ごとに「URL・最終確認日」の一覧を作る
- 引数で章が指定されていればその章だけ、指定がなければ古い日付（90日以上前）を優先して全章を対象にする

### 2. URLの生存と内容の確認

各URLについて WebFetch で実際に開き、次を確認する。

| 確認項目 | 判定 |
| ---- | ---- |
| HTTP到達性 | リダイレクト先が公式の後継ページなら、URL自体を差し替える |
| 本文との整合 | そのURLを根拠にしている本文の記述（機能名・モデル名・プラン・料金）がまだ正しいか |
| 製品名の変更 | 改名・統合（例: サービス名のリブランド）があれば、本文側の修正候補として記録する |

ボリュームが多い場合は、URL確認をサブエージェント（Explore または general-purpose、modelはsonnet）に章単位でファンアウトしてよい。**ただし日付更新の判断は必ず確認結果を受けてから行う。**

### 3. 周辺の新着情報の巡回（オプション）

棚卸しに加えて「最新の話題も拾って」と頼まれた場合のみ、WebSearch で次の定点を確認する。

- Anthropic: anthropic.com/news、platform.claude.com/docs の更新
- Google: blog.google（AI関連）、Workspace Updates blog
- OpenAI: openai.com/blog
- プライバシー・規約系: 各社のプライバシーポリシー更新告知、個人情報保護委員会
- 日本語圏の反応: Publickey、ITmedia など（一次ソースの補助として）

拾った話題は「どの章の文脈に効くか」を添えて報告し、本文への反映は利用者の指示を待つ。

### 4. 修正の適用

- リンク切れ・後継URLへの差し替え: 参考エントリを書き換え、最終確認日を当日に更新する
- 本文の陳腐化: WRITING_GUIDE.md のトーン（です・ます調、断定回避、節タイトル＝結論）に従って該当段落を書き直す。語句置換ではなく節単位で読み直す
- 確認したが変更不要だったURL: 最終確認日のみ当日に更新する
- 開けなかった・確認しきれなかったURL: 日付を**動かさず**、レポートに残す

### 5. 仕上げ

```bash
npm install && npm run lint
```

- lint警告は原則解消する（prh-style.yml の検出は注意喚起なので、機械置換せず文脈で判断）
- コミットは章単位。PRを作る場合は「docs: 参考URLの棚卸し（YYYY-MM）」の体裁で、本文に「確認したURL数／差し替え件数／本文修正件数／未確認で残した件数」を書く

## レポートの形式

巡回の最後に、次の表を必ず出す。

| 章 | 確認したURL | 結果 | 対応 |
| ---- | ---- | ---- | ---- |
| 13章 | Anthropic Models overview | 内容更新あり（新モデル追加） | 本文修正＋日付更新 |
| 09章 | Anthropic Privacy Policy | 変更なし | 日付のみ更新 |
| 02章 | （例）リンク切れ | 404 | 後継URLに差し替え |

未確認のまま残したURLと、人の判断が要る論点（記述を変えるか迷うもの）は別枠で列挙する。
